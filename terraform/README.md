# Terraform — AKS for Dev & Test (POC)

Provisions cost-efficient, enterprise-shaped AKS environments on Azure for the
Employee Management System. One reusable root module + three child modules,
driven by a per-environment `.tfvars` file. **Dev** and **Test** differ only in a
few cost knobs.

```
terraform/
├── versions.tf providers.tf variables.tf main.tf outputs.tf backend.tf
├── dev.tfvars           # dev cost profile
├── test.tfvars          # test cost profile
└── modules/
    ├── network/         # VNet + AKS subnet + NSG
    ├── keyvault/        # RBAC Key Vault + secrets + role assignment
    └── aks/             # AKS cluster (overlay + Calico, workload identity, KV CSI)
```

## What gets created (per environment)

| Resource | Notes |
| --- | --- |
| Resource Group | `rg-emp-<env>` |
| Virtual Network + Subnet + NSG | `10.10.0.0/16` (dev) · `10.20.0.0/16` (test) |
| AKS cluster | Free tier, 1 system node pool, **Azure CNI Overlay + Calico** |
| Key Vault (RBAC) | stores `jwt-secret`, `postgres-password`, `seed-admin-password` |
| Azure Container Registry | Basic, attached to AKS (`AcrPull`) — optional |
| Log Analytics | test only (Container Insights) — optional |
| Role assignments | deployer→KV Secrets Officer, AKS CSI→KV Secrets User, kubelet→AcrPull |

Secrets are **generated by Terraform** (`random_password`) and pushed to Key
Vault — never hardcoded. The Key Vault Secrets Provider (CSI) add-on lets pods
mount them via Workload Identity.

## Dev vs Test (the only differences)

| | Dev | Test |
| --- | --- | --- |
| Node VM | `Standard_B2s` (2 vCPU / 4 GiB) | `Standard_B2ms` (2 vCPU / 8 GiB) |
| Cluster autoscaler | off (fixed 1 node) | on (1 → 3 nodes) |
| Container Insights | off | on |
| Address space | `10.10.0.0/16` | `10.20.0.0/16` |

Everything else (Free control plane, single system pool, small OS disk, Basic
ACR, overlay networking) is identical. Verified with `terraform plan`:
**dev = 19 resources, test = 20** (the extra one is Log Analytics).

## Cost efficiency choices

- **Free** control-plane tier (no uptime-SLA charge).
- **Burstable B-series** VMs and a small **32 GiB** OS disk.
- **Single** system node pool (no always-on user pool); dev runs just 1 node.
- **Azure CNI Overlay** → pod IPs don't consume VNet address space.
- **Basic** ACR; **Standard** Key Vault.
- Container Insights only in test (log ingestion is the usual cost surprise).
- `Free` tier + 1 small node keeps dev to roughly a few dollars/day; stop with
  `terraform destroy` when idle.

## State separation (Option B — one state object per environment)

Dev and Test keep **completely separate state files** in the same Azure Storage
container, keyed by environment. Nothing in the `.tf` files hardcodes the
environment, so state isolation is what keeps the two apart:

```
Storage container "tfstate"
  ├── dev.tfstate     ← selected by -backend-config=key=dev.tfstate
  └── test.tfstate    ← selected by -backend-config=key=test.tfstate
```

The backend block in `backend.tf` is empty on purpose — all settings are
injected at `terraform init` time, so switching environment = re-init with a
different `key`. The `make tf-*` targets always pair the correct `key=<env>`
with `<env>.tfvars`, so you can't mismatch them.

### 1. One-time: create the remote state storage

```bash
make tf-bootstrap        # az login first; creates rg-tfstate-emp + storage acct + container,
                         # then writes terraform/backend.hcl (safe to commit — no secrets)
```

### 2. Plan / apply per environment

```bash
make tf-plan  TF_ENV=dev     # init -reconfigure -backend-config=key=dev.tfstate,  then plan
make tf-apply TF_ENV=dev

make tf-plan  TF_ENV=test    # init -reconfigure -backend-config=key=test.tfstate, then plan
make tf-apply TF_ENV=test
```

Equivalent raw commands (what the targets run):

```bash
cd terraform
terraform init -reconfigure -backend-config=backend.hcl -backend-config=key=dev.tfstate
terraform apply -var-file=dev.tfvars
# switch env == re-init with a different key:
terraform init -reconfigure -backend-config=backend.hcl -backend-config=key=test.tfstate
terraform apply -var-file=test.tfvars
```

> Because each env has its own state object, applying `test` can never touch
> `dev`'s resources, and state locking is per-object (concurrent dev/test
> applies don't block each other). No workspaces needed.

### 3. Connect and deploy the Phase 3 Helm chart

```bash
# Terraform prints the exact command:
terraform output -raw aks_get_credentials_command | bash

# push images to the new ACR
ACR=$(terraform output -raw acr_login_server)
az acr login --name "${ACR%%.*}"
# docker tag / push employee-management/* -> $ACR/*

# deploy — secrets come straight from Key Vault (no hardcoded secrets):
helm upgrade --install ems ../helm/employee-management \
  -n employee-dev --create-namespace -f ../helm/employee-management/values-dev.yaml \
  --set image.registry=$ACR \
  $(terraform output -raw helm_keyvault_set_flags)
```

Because AKS uses **Azure CNI Overlay + Calico**, the Phase 3 NetworkPolicies are
actually **enforced** here (unlike local kind/kindnet).

## Consuming Key Vault secrets from pods — Workload Identity (wired into the chart)

Secrets are **never** hardcoded. The chart ships a `SecretProviderClass` that the
Secrets Store CSI driver uses to pull `jwt-secret` / `postgres-password` /
`seed-admin-password` from Key Vault and sync them into a Kubernetes Secret the
pods read as env vars (`DATABASE_URL` is assembled in-pod from the parts).

Auth uses **Workload Identity** (least privilege — no node-wide identity):
- Terraform creates a dedicated **user-assigned identity** (`id-emp-<env>-kv`),
  grants it **Key Vault Secrets User**, and federates it to the app's Kubernetes
  ServiceAccount via a `azurerm_federated_identity_credential`
  (`subject = system:serviceaccount:employee-<env>:ems-employee-management`).
- The chart annotates that ServiceAccount with the UAMI client ID and labels the
  backend/postgres pods `azure.workload.identity/use: "true"`; the CSI driver
  exchanges the pod's federated SA token for a Key Vault token.

`terraform output helm_keyvault_set_flags` prints the exact `--set keyVault.*`
flags (vault name, tenant, UAMI client ID) — pass them to `helm upgrade`.

> The federation is bound to a specific namespace + ServiceAccount. If you change
> the Helm **release name** or **namespace**, set `app_namespace` /
> `app_service_account` in the tfvars to match (the SA name is
> `<release>-employee-management`). Rotate a secret by updating Key Vault — the
> CSI driver refreshes it (2-minute poll).

## State backend hardening (already applied)

`scripts/bootstrap-state.sh` provisions the state storage account with TLS 1.2
minimum, public blob access disabled, and blob **versioning + 7-day soft
delete** (so a bad apply can be rolled back). For stricter setups, disable
shared-key access and switch the backend to `use_azuread_auth = true` with a
`Storage Blob Data Contributor` role for the deployer.

## Tear down

```bash
make tf-destroy TF_ENV=dev
make tf-destroy TF_ENV=test
# optionally remove the state storage too:
az group delete -n rg-tfstate-emp --yes
```
